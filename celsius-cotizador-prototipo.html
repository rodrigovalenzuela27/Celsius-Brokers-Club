§ Mecánica del apartado

Al generar la cotización, la unidad 805 queda apartada por 24 horas en el inventario maestro. Durante ese tiempo:

  • Ningún otro broker puede generar cotización sobre la misma unidad.
  • Se crea automáticamente un Deal en HubSpot en etapa "Apartado 24h".
  • Se envía PDF al cliente y notificación al broker.
  • Si no se confirma reserva formal en 24h, la unidad se libera automáticamente.

§12 · Arquitectura del sistema · v0.1

Arquitectura técnica

Stack recomendado y diagrama de capas. Event-driven: HubSpot es CRM maestro, la plataforma es el sistema de inventario y cotización, ambos se sincronizan vía webhooks.

Stack recomendado

FRONTENDNext.js 15 · React 19 · TypeScript · Tailwind
UI KITshadcn/ui personalizado con tokens Celsius
APINext API Routes + tRPC (typesafe end-to-end)
DBPostgreSQL gestionado en Supabase (RLS + auth)
AUTHSupabase Auth (email/pass + SSO HubSpot opcional)
STORAGESupabase Storage (PDFs, brochures, planos, renders)
CACHE / LOCKSUpstash Redis (apartados 24h con TTL automático)
QUEUE / JOBSInngest o Vercel Cron (liberar apartados, sync nocturno)
PDFReact-PDF o Puppeteer headless
EMAILResend (transaccional)
FIRMAMifiel (México) o DocuSign
ANALYTICSPostHog (producto) + Supabase (operación)
HOSTINGVercel (frontend/API) · Supabase Cloud (DB)
OBSERVABILIDADSentry · Axiom (logs estructurados)

Por qué este stack

Optimizado para velocidad de construcción con Claude Code, costo bajo en MVP, y capacidad de escalar a multi-desarrolladora sin rediseñar.

  • Next.js + Supabase permite que un solo desarrollador (o tú con Claude Code) cubra frontend, API, DB y auth desde el día uno.
  • tRPC elimina el contrato manual entre frontend y backend — Claude Code genera ambos lados consistentes.
  • Upstash Redis + TTL es la forma más limpia de implementar el apartado de 24h sin cron-jobs frágiles.
  • HubSpot como CRM maestro permite a tu equipo comercial seguir trabajando en HubSpot mientras la plataforma alimenta el funnel.
  • Mifiel es estándar en México para firmas electrónicas de apartados y promesas de compra.
  • Todo el stack tiene tier gratis o muy bajo costo en MVP (< USD 100/mes los primeros meses).

§13 · Modelo entidad-relación · v0.1

Modelo de datos

Entidades principales del MVP. Diseñado para soportar multi-desarrolladora desde el día uno (campo developer_id en cada tabla) sin sobre-ingeniería.

Notas de diseño

  • developer_id en cada entidad maestra prepara el camino para multi-desarrolladora sin migración futura.
  • UNIT_HOLD con constraint unique en unit_id garantiza que una unidad sólo puede tener un apartado activo a la vez (a nivel DB).
  • expires_at es la fuente de verdad — un job cron/Inngest lo revisa cada minuto y libera apartados expirados.
  • hubspot_contact_id y hubspot_deal_id son los punteros para reconciliación bidireccional.
  • svg_coords en UNIT guarda las coordenadas del polígono dentro del SVG del edificio — esto es lo que permite el visualizador interactivo.
  • AUDIT_EVENT registra cada cambio para trazabilidad legal/operativa (quién cotizó qué a qué precio y cuándo).
§14 · Flujos de usuario · proceso end-to-end

Flujos clave

Tres flujos principales: alta de cliente, cotización con apartado, y resolución del apartado (compra o liberación automática).

Flujo 2 — Resolución del apartado en 24h

Flujo 3 — Sincronización bidireccional con HubSpot
